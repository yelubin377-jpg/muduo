```c++
～锁竞争
时间顺序 →

  ① 线程B 执行 if(x)      →  x 不是 NULL，✓ 通过检查
  ② 线程A 执行 delete x   →  对象析构，mutex_ 随之销毁
  ③ 线程B 执行 update()   →  💥 对象已死，程序崩溃

  ① 和 ② 的顺序是不确定的（竞态）：
  如果 ② 插在 ① 和 ③ 之间，就崩了。
```

```c++
～锁竞争
Foo::~Foo()
{
    MutexLockGuard lock(mutex_);
}
void Foo::update()
{
    MutexLockGuard lock(mutex_);
}

//thread A
delete x;
x = NULL;

//thread B
if(x)
{
    x->update(); // 会调用锁，但是锁已经被销毁了      ->这里程序有概率崩溃
}
```

```c++
～临界区重叠问题
时间顺序 →

  ① 线程A 拿到锁   →  进入临界区，执行 value_++
  ② 线程B 尝试拿锁  →  被挡在外面，等待...
  ③ 线程A 释放锁   →  退出临界区
  ④ 线程B 拿到锁   →  进入临界区，执行 value_++

  同一时刻只有一个线程在临界区里——这是锁的正常工作。
```

> ​	～临界区重叠:
>
> ​		两线程同时进入保护同一数据临界区
>
> ​		线程A：刚拿到锁，执行value_ ++;  ->     数据
>
> ​		线程B：同A                                         ->     竞争
